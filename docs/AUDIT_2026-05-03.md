## 🔴 CRITICAL — bugs reales

### 1. Spoiler residual: `ProjectionWaiting` muestra candidatas de la ronda actual
- **Dónde:** `src/hooks/useProjectionData.ts:348-353` (cálculo `previouslySelected`) + `src/components/projection/ProjectionWaiting.tsx:169` (`<SelectedCandidatesSidebar candidates={previouslySelected} />`).
- **Qué pasa:** `previouslySelected` se construye como `[...selectedCandidates].sort(...)`, donde `selectedCandidates = candidates.filter(c => c.is_selected)`. **No filtra por `selected_in_round < current_round_number`**. Si una ronda se finaliza con `round_finalized=true` pero `show_results_to_voters=false`, el `waitingMode` pasa a `"finalized"` y la sidebar derecha sigue mostrando las candidatas seleccionadas en la ronda actual → spoiler antes de que el admin pulse "Ver resultados".
- **Por qué importa:** rompe el dramatismo del reveal y destruye el flujo "finalizar → anunciar → mostrar".
- **Propuesta:**
  ```ts
  // useProjectionData.ts:348
  const currentRn = round?.current_round_number ?? Number.POSITIVE_INFINITY;
  const previouslySelected = selectedCandidates
    .filter(c => (c.selected_in_round ?? Number.POSITIVE_INFINITY) < currentRn)
    .sort((a, b) => {
      const ra = a.selected_in_round ?? 999;
      const rb = b.selected_in_round ?? 999;
      if (ra !== rb) return ra - rb;
      return (b.selected_vote_count ?? 0) - (a.selected_vote_count ?? 0);
    });
  ```

### 2. Barra "personas en sala" usa 100 como denominador fijo
- **Dónde:** `src/components/projection/ProjectionWaiting.tsx:165`.
- **Código:** `<div className="proj-stat-progress-fill" style={{ width: \`${Math.min(connectedCount, 100)}%\` }} />`
- **Qué pasa:** la barra de ocupación nunca usa `max_votantes` de la ronda. Con `max_votantes=200` y 50 conectados marca 50% (debería ser 25%). Con `max_votantes=20` y 50 conectados marca 50% (debería estar al 100% saturada).
- **Por qué importa:** dato proyectado en pantalla grande → da impresión falsa de aforo al público.
- **Propuesta:** propagar `maxVotantes` desde `useProjectionData` (ya lo tiene en `round.max_votantes`) hasta `ProjectionWaiting` y calcular:
  ```ts
  const pct = round?.max_votantes && round.max_votantes > 0
    ? Math.min(100, Math.round((connectedCount / round.max_votantes) * 100))
    : 0;
  ```

### 3. `unselectCandidate` reabre rondas con escritura no atómica desde cliente
- **Dónde:** `src/components/admin/AdminVotingDetail.tsx:655-687`.
- **Qué pasa:** después de `unselect_candidate` RPC, el cliente vuelve a leer `candidates`, recuenta `is_selected` y, si baja del cupo, hace UPDATE directo a `rounds` modificando 5 flags simultáneos (`is_closed:false, is_active:true, round_finalized:true, show_results_to_voters:false, show_ballot_summary_projection:false`). Entre la lectura y el UPDATE puede:
  - Otro admin/RPC seleccionar candidatas de nuevo → reapertura indebida
  - Fallar parcialmente y dejar `rounds` inconsistente (ej. `is_active:true` con `round_finalized:false` por trigger ajeno)
- **Por qué importa:** drift de estado en la tabla `rounds`, reapertura de votaciones que el admin creía cerradas.
- **Propuesta:** mover toda la decisión a un RPC server-side `reopen_round_after_unselect(p_candidate_id, p_round_id)` que en una transacción haga el unselect + recálculo + UPDATE condicional con `WHERE selected_count < max_selected`.

### 4. `confirmSelection` actualiza 7 columnas de `rounds` sin transacción
- **Dónde:** `AdminVotingDetail.tsx:400-419`.
- **Qué pasa:** un solo `UPDATE` (que sí es atómico a nivel SQL) pero sin guardas server-side: el cliente decide cuándo se "confirma" sin verificar que `selected_count >= max_selected`. Si el cliente se equivoca (selección obsoleta en estado, race con realtime), se cierra la votación sin cupo cumplido.
- **Por qué importa:** confirmación prematura → galería final con plazas vacías o duplicadas.
- **Propuesta:** RPC `confirm_round_selection(p_round_id)` que valide `(SELECT count(*) FROM candidates WHERE round_id=$1 AND is_selected) >= max_selected_candidates` antes de cerrar.

### 5. `useEffect` de `VotingPage` re-suscribe canal en cada cambio de `loadActiveRound`
- **Dónde:** `src/pages/VotingPage.tsx:526` (deps `[navigate, loadActiveRound, toast, previewMode]`).
- **Qué pasa:** `loadActiveRound` está envuelto en `useCallback` con deps `[toast, loadResults, ensureSeat, loadStoredReceipt]`. `toast` viene de `useToast()` que **no garantiza referencia estable** entre renders en todas las versiones; `ensureSeat` y `loadStoredReceipt` son también `useCallback`. Cada vez que cualquiera cambie de identidad, el efecto cleanup ejecuta `removeChannel` y vuelve a `subscribe`, generando ráfagas de subscripciones realtime.
- **Por qué importa:** consumo de cuota Supabase realtime, posibles `SUBSCRIBED` huérfanos, picos de CPU en cliente.
- **Propuesta:**
  - Crear `useLatestRef(toast)` y usar `toast.current(...)` dentro del callback para no incluir `toast` en deps.
  - Sacar `loadActiveRound` del scope de hooks o memoizarlo con deps reales mínimas.
  - Idealmente: crear `useVoterRoundSubscription(onChange)` separado que monte UNA vez con `[]` y dispare callback estable.

### 6. Canal admin sin sufijo único
- **Dónde:** `AdminVotingDetail.tsx:257` — `supabase.channel(\`admin-voting-detail-${roundId}\`)`.
- **Qué pasa:** dos pestañas del mismo admin abriendo la misma ronda generan canales con nombre idéntico. Supabase rechaza el segundo silenciosamente; la pestaña que perdió la carrera se queda sin realtime y solo recibe updates por el `setInterval(30s)`.
- **Propuesta:**
  ```ts
  const uid = useRef(crypto.randomUUID()).current;
  supabase.channel(`admin-voting-detail-${roundId}-${uid}`)
  ```

## Respuesta a preguntas del graphify

### "Why does `toast()` connect Admin Voting Detail to Device Identity & Fingerprinting?"
`toast()` es **el bridge correcto** — es el sistema de feedback compartido. 32 INFERRED edges son válidos. No es un problema, es el patrón esperado: cualquier acción admin/votante notifica vía toast. Lo único accionable: usar `useToast` con ref estable para no re-suscribir efectos (#5 CRITICAL).

### "Should Admin Voting Detail be split?" (cohesión 0.07)
**Sí.** Ver HIGH #2. La cohesión bajísima refleja que el archivo mezcla 8 responsabilidades. Plan de split incluido.

### "Why does `submitVote()` bridge Device Identity into Admin?"
`submitVote` usa `generateDeviceHash`, `generateBrowserInstanceId`, `markAsVoted` (community Device Identity) + `cast_ballot` RPC + lectura de `rounds` (community Admin). El bridge es real porque la lógica de voto **mezcla cliente (fingerprint) con dominio admin (rounds/votes)**. Lo correcto sería extraer `useVoteSubmission()` (HIGH #3) que encapsule ambos lados.

### "Why is `ComunicaImport` isolated?"
Feature aislada por diseño (route-only). Ver LOW #25. No requiere acción.

## Investigación: imágenes desde ComunicaImport (SinergiaCRM)

### Pregunta
¿Se pueden importar/descargar las fotos de los contactos del CRM al importar candidatos?

### Conclusión: **Factible — alta viabilidad**

**Cómo se construye la URL de foto:**
```
https://{dominio_crm}/index.php?entryPoint=download&id={crm_id}_photo&type=Contacts
```
El `crm_id` ya se obtiene en cada contacto (`raw['id']`). El sufijo `_photo` es fijo. El dominio base se extrae de `SINERGIA_URL`.

**Por qué no funciona como `<img src="...">` directo:**
El endpoint `entryPoint=download` de SugarCRM/SinergiaCRM requiere sesión autenticada. Sin cookie de sesión el servidor devuelve 302 o 401. No se puede usar como URL pública directa.

**Solución propuesta (F-PHOTOS):**
1. **crm-proxy** — nueva acción `fetch-photos`:
   - Input: `contact_ids: string[]`
   - Re-autentica con CRM (reutiliza session si se pasa)
   - Para cada ID: `GET {CRM_BASE}?entryPoint=download&id={id}_photo&type=Contacts` con `Cookie: PHPSESSID=...` del login
   - Sube imagen a Supabase Storage bucket `candidate-photos/{round_id}/{crm_id}.jpg`
   - Devuelve `{ [crm_id]: public_url | null }`
   - Manejo graceful: si foto no existe (204/404), devuelve `null` para ese ID
2. **Supabase Storage**: bucket `candidate-photos` público (lectura)
3. **ComunicaImport.tsx** — paso opcional post-importación "Importar fotos":
   - Llama `fetch-photos` con los `crm_id` importados
   - Muestra progreso con contador
   - Actualiza `image_url` en DB para cada candidato
4. **AdminVotingDetail** — botón "Re-importar fotos desde CRM" para candidatos ya existentes
5. **Estimación esfuerzo:** ~4 h (1h crm-proxy + 0.5h storage + 2h UI + 0.5h tests)

**Riesgo:** La autenticación de SugarCRM usa `PHPSESSID` cookie vía login REST v4.1 (`login` devuelve session id). Hay que verificar que el proxy puede construir una cookie de sesión válida. Alternativa: algunos SugarCRM aceptan `?sugar_token={session_id}` como query param en lugar de cookie.
