## 🔴 CRITICAL

### 1. Vote receipt persiste en localStorage antes de confirmar el INSERT en BD
- **Dónde**: `src/pages/VotingPage.tsx:772-795`.
- **Qué pasa**: `persistVoteReceipt(persistedReceipt)` se llama (línea 779)
  ANTES de ejecutar el `supabase.from('votes').insert(votes)` (línea 793).
  Si el INSERT falla (red, RLS, duplicate), el ticket queda guardado en
  localStorage como si hubiera votado. En la siguiente recarga
  `loadStoredReceipt` lo recupera y muestra ticket de un voto que no existe
  en BD.
- **Por qué importa**: Falsa confirmación al votante; auditoría inconsistente
  entre cliente y servidor.
- **Propuesta**: Mover `persistVoteReceipt(...)` después de comprobar
  `if (voteError) return;` (línea 798). El `voteCode` ya está calculado, no
  hay riesgo.

### 2. `markAsVoted` se llama incluso si el INSERT falla parcialmente
- **Dónde**: `src/pages/VotingPage.tsx:809`.
- **Qué pasa**: `markAsVoted` se ejecuta inmediatamente después del insert,
  pero si el array `votes` se inserta parcialmente (Supabase no es transaccional
  en INSERT múltiple por defecto sin trigger), se marca como votado aunque
  falten filas.
- **Por qué importa**: papeleta inconsistente y bloqueo de reintento del usuario.
- **Propuesta**: Ejecutar el insert múltiple dentro de un RPC server-side
  `cast_ballot(p_round_id, p_seat_id, p_candidate_ids[])` que lo envuelva en
  transacción, devolviendo `success bool, vote_hash text`.

### 3. `quickSelectCandidate` y `forceSelectCandidate` no usan RPC y descuadran `selected_candidates_count`
- **Dónde**: `AdminVotingDetail.tsx:556-576` (force) y `731-744` (quick).
- **Qué pasa**: ambos hacen `UPDATE candidates SET is_selected=true` directo y
  `forceSelectCandidate` además hace `UPDATE rounds SET selected_candidates_count = selectedCandidatesCount + 1`
  con un valor *cliente*. Si entre la lectura y el update otro admin hizo cambios,
  el contador queda desincronizado. `quickSelectCandidate` ni siquiera actualiza
  el contador.
- **Por qué importa**: `selected_candidates_count` es la base de
  `selectionQuotaReached`, que cierra rondas. Drift = ronda que no se cierra
  o se cierra antes de tiempo.
- **Propuesta**: Crear RPC `force_select_candidate(p_candidate_id)` server-side
  con `UPDATE rounds SET selected_candidates_count = (SELECT count(*) FROM candidates WHERE round_id=... AND is_selected=true)`
  atómico tras marcar la fila. Reusar lo mismo en `quickSelectCandidate`.

### 4. `confirmSelection` cierra la votación incluso sin alcanzar `max_selected_candidates`
- **Dónde**: `AdminVotingDetail.tsx:435-454`.
- **Qué pasa**: La función solo comprueba `round_finalized`; setea
  `is_closed:true, public_candidates_enabled:false`. Si el admin pulsa
  "Confirmar selección" en el primer ciclo del workflow (label puede llegar
  ahí cuando `selectionQuotaReached`), pero también en otros caminos, puede
  cerrar la ronda con menos seleccionadas que `max_selected_candidates`.
- **Por qué importa**: cierre prematuro irreversible (lista pública desactivada
  permanentemente, según el guard del `togglePublicCandidates`).
- **Propuesta**: Añadir guard explícito al inicio:
  ```ts
  if (!selectionQuotaReached) {
    toast({ title: 'No se ha alcanzado el cupo', description: `Faltan ${maxSelectedCandidates - selectedCandidatesCount}.`, variant: 'destructive' });
    return;
  }
  ```

### 5. Falta dependencia `[candidates]` en `ensureSeat` causa fingerprint desactualizado
- **Dónde**: `VotingPage.tsx:293-355`. `useCallback(..., [])` (línea 355) — dependencia vacía.
- **Qué pasa**: `ensureSeat` no depende de nada externo cambiante porque solo
  recibe `round`, OK. Pero `loadActiveRound` (línea 393) tiene `useCallback(..., [toast, loadResults, ensureSeat, loadStoredReceipt])`.
  `loadActiveRound` se referencia dentro de un `useEffect` con `[navigate, loadActiveRound, toast, previewMode]`. Cada vez que `ensureSeat`
  se recree (no debería pero si hay refactor), se desuscribe y reconnecta el realtime,
  lo que puede dejar al usuario sin actualizaciones momentáneas.
- **Por qué importa**: Bajo, pero real: durante recargas de subscripción se
  pueden perder cambios de `is_voting_open`.
- **Propuesta**: extraer la lógica de subscripción a su propio `useEffect` con
  `[]` y usar `activeRoundRef` dentro. (Hoy ya se intenta, pero `loadActiveRound`
  está en deps.)

### 1. Tres sistemas de estilos coexisten — propuesta de unificación
- **Dónde**: ver conteo:
  - `style={{` inline → 870 ocurrencias en 29 archivos.
  - clases `avd-*` → 37 ocurrencias declaradas (más usos).
  - Tailwind utilities → toda `VotingPage.tsx` usa `text-foreground`,
    `bg-surface-container-lowest`, `flex`, `gap-3`, etc.
- **Qué pasa**: cada pantalla principal usa un dialecto distinto:
  - **Admin** (`AdminVotingDetail`, `AdminVotingList`, `TeamChip`): clases
    `avd-btn`, `avd-chip-*`, `avd-input` con tokens `var(--avd-*)`.
  - **Projection** (`ProjectionVoting`, `Results`, `Waiting`, `_shared.tsx`):
    todo `style={{}}` inline con los mismos tokens `var(--avd-*)`.
  - **Voting** (`VotingPage` 1744 líneas): mezcla brutal — Tailwind para los
    `flex`/`text-*`, `style={{}}` inline para tickets, y `avd-chip` cuando
    quiere reutilizar.
- **Por qué importa**: no se puede cambiar un radio o un color sin tocar 4
  sitios. Cada pantalla "se ve distinta" porque son sistemas paralelos.
- **Propuesta — fases concretas**:
  1. **Token único**: las variables `--avd-*` ya están en CSS; añadir mapeo
     en `tailwind.config.ts` para que `bg-brand`, `text-fg`, `border-default`
     resuelvan a `var(--avd-brand)` etc. Coste: 1 archivo.
  2. **Migrar `_shared.tsx` a clases**: convertir `chipStyle()`, `card()`,
     `AccentBar` a componentes con `className=` usando las clases `avd-*`
     que ya existen. Eliminar las 174 líneas de inline styles del archivo.
  3. **Projection inline → clases**: `ProjectionVoting/Results/Waiting/Final`
     son el bloque más limpio (solo `style={{}}`). Crear `projection.css` con
     clases `proj-header`, `proj-stat-big`, `proj-vote-count` y reemplazar.
     Estimación: 4-6h (≈900 líneas inline a sustituir).
  4. **VotingPage**: el monstruo. Recomendación: dejar Tailwind donde ya está
     y migrar solo los grandes bloques inline (ticket de voto, gracias) a
     componentes en `src/components/voting/VoteTicket.tsx`. Usa clases
     `avd-*` allí.
  5. Una vez todo sea `className=`, eliminar `style={{}}` inline excepto
     casos genuinamente dinámicos (anchos de barras `width:${pct}%`,
     `animationDelay`).

  `count: 'exact'` sobre `votes` filtrando por `round_number`, en lugar de
  confiar en el trigger. O incrementar localmente con `setVoteCount(p => p+1)`
  en cada evento + reconciliar al recargar.

- **Qué pasa**: cierre directo sin desactivar lista pública (`public_candidates_enabled`),

- **Propuesta**: si `selectedCandidatesCount > 0`, redirigir a confirmar

- **Dónde**: `useProjectionData.ts:301-306` (poll 5s) y
  `AdminVotingDetail.tsx:295` (realtime seats).
- **Qué pasa**: en projection se polea cada 5s aunque ya hay realtime de

  consume requests cuando la sala está vacía.
- **Propuesta**: en projection, suscribirse a realtime de `seats` con
  `filter: round_id=eq.${round.id}` y eliminar el polling.

### 4. `metricsInterval` 10s en `AdminVotingDetail` con realtime activo
- **Dónde**: `AdminVotingDetail.tsx:298-301`.
- **Qué pasa**: el realtime cubre `votes` (línea 296). El interval recarga

  realtime + recarga manual al abrir el detail.

### 5. Tipos duplicados `Round`/`Candidate` en 5 archivos
